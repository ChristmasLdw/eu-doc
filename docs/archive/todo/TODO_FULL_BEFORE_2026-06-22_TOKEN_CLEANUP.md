# eu-doc 三阶段产品与技术 TODO（AI 后续工作规划）

最后更新：2026-06-22 CST  
规划目的：帮助后续 AI 理解用户真实意图，避免只做表面功能，优先完成会影响长期扩展的产品地基。

## 0. 用户当前策略

用户当前不是马上商业化，而是按三阶段推进：

1. **第一阶段：个人测试 / 本地开发**
   - 在用户自己的电脑上开发和测试。
   - 用户自己模拟真实企业录入资料。
   - 目标是打磨基础功能、验证流程、避免后续推广时出现严重体验问题。

2. **第二阶段：测试服务器 + 小域名 + 小范围企业测试**
   - 转移到腾讯云服务器和小域名。
   - 允许外部用户访问。
   - 允许约 10-20 家真实企业注册、认证、上传资料。
   - 目标是验证真实企业对基础功能的使用体验。

3. **第三阶段：正式服务器 / 正式域名 / 推广商业化**
   - 转移到更大的服务器和正式域名。
   - 面向更多企业和全球用户。
   - 再逐步开启收费、支付、套餐、API、OCR、数据分析等商业化能力。

核心原则：

```text
可以后移的功能不要急着做。
如果后移会导致数据迁移、权限重构、页面大改，就必须现在先做扎实。
```

## 1. 项目长期定位

`eu-doc` 最终要变成一个平台，而不是单企业展示站。

长期定位：

```text
eu-doc 是一个面向全球企业、消费者、采购商、经销商和审批机构的产品合规资料平台。
```

平台核心内容包括：

- 产品认证证书
- DoC 声明文件
- 产品使用说明书
- 检测报告
- 其他合规资料

平台核心价值：

- 可信
- 可查
- 可验证
- 可追溯
- 方便企业维护
- 方便外部用户查看
- 方便审批机构查验 DoC 和合规文件

架构方向必须从：

```text
企业 -> 证书
```

升级为：

```text
企业 -> 产品 -> 文档
```

其中：

```text
文档 = 证书 + DoC + 说明书 + 检测报告 + 其他合规资料
```

这是后续所有开发的核心方向。

## 2. 功能前置 / 后移判断

### A 类：现在必须做扎实

这些属于长期地基。现在不做，后续真实企业数据进入后会很难改。

| 功能 | 是否现在做 | 原因 |
|---|---:|---|
| 产品模型 `products` | 必须现在做 | 平台本质是产品资料库，不是单纯证书库 |
| 文档模型 `documents` | 必须现在做 | 证书、DoC、说明书、检测报告不能都塞进证书表 |
| 分类体系 `categories` | 必须现在做 | 后期数据多了再改分类会影响搜索、筛选、后台、统计 |
| 标签体系 `tags` | 必须现在做 | 支持跨行业、跨标准、跨用途筛选 |
| 用户模型 `users` | 必须现在做 | 当前 `admins` 思路不适合平台化 |
| 企业成员 `company_members` | 必须现在做 | 第二阶段企业员工权限会用到 |
| 企业认证状态 | 必须现在做 | 第二阶段真实企业上传前必须有认证状态 |
| 上传确认 / 免责声明记录 | 必须现在做 | 涉及责任链，不能后补 |
| 审计日志 | 必须现在做 | 后续追责、排查、审核都依赖它 |
| 环境变量配置 | 必须现在做 | 项目会经历本地、测试域名、正式域名迁移 |
| 备份结构 | 必须现在做 | 第二阶段开始会产生真实企业数据 |
| 多语言字段预留 | 必须现在做 | 最终面向全球，后补多语言成本高 |

### B 类：第二阶段前必须可用

这些第一阶段可以做简版，但在给 10-20 家企业测试前必须稳定可用。

| 功能 | 第一阶段状态 | 第二阶段前要求 |
|---|---|---|
| 真实注册登录 | 可先简化 | 必须稳定可用 |
| 邮箱验证 | 可先模拟 | 必须真实可用 |
| 企业认证 | 可先手动 | 必须能提交和审核 |
| 企业成员权限 | 可先基础角色 | 必须支持邀请或分配权限 |
| 企业上传资料 | 第一阶段模拟 | 第二阶段真实企业可上传 |
| 外部用户访问 | 可本地测试 | 小域名必须稳定可访问 |
| HTTPS | 可暂不做 | 第二阶段必须有 |
| 帮助中心 | 可先草稿 | 第二阶段必须指导用户 |
| 法律入口 | 可先简版 | 第二阶段必须有入口 |
| 举报 / 反馈 | 可先简单表单 | 第二阶段必须可用 |
| 管理员审核后台 | 可先基础 | 第二阶段必须能审核企业和资料 |
| 数据导出 | 可后置一点 | 第二阶段最好有基础导出 |

### C 类：第三阶段再做，但现在要预留接口

这些不需要现在做完整，但数据结构和代码设计不要堵死。

| 功能 | 现在要做什么 | 第三阶段做什么 |
|---|---|---|
| 支付系统 | 预留套餐、订阅、用量表 | 接微信、支付宝、Stripe、PayPal |
| 套餐系统 | 预留 plan 字段和权限判断入口 | 做套餐页和购买流程 |
| 用量统计 | 先记录产品数、文件数、存储量 | 用于收费和限制 |
| OCR | 预留 `ocr_status`、`extracted_text`、`metadata_json` | 接 OCR 服务 |
| API 开放 | 接口命名保持规范 | 做 API Key 和开发者文档 |
| 对象存储 / CDN | 文件路径不要写死 | 接腾讯云 COS / CDN / S3 |
| 第三方企业核验 | 保留认证状态和资料表 | 接工商核验接口 |
| 数据分析 | 先记录访问、搜索、下载 | 做企业数据报表 |
| 自定义域名 | 不硬编码当前域名 | 企业版支持绑定域名 |

### D 类：可以更晚做

这些现在不用投入太多：

- SSO 单点登录
- 白标部署
- 电子签章
- 复杂组织架构
- 多区域容灾
- AI 自动审核
- 移动 App
- 开发者平台
- 大型企业 SLA

## 6. 推荐开发顺序

### 第一优先级：先做结构，不急着美化

```text
1. products
2. documents
3. categories
4. tags
5. users
6. company_members
7. company verification
8. upload confirmations
```

原因：这些决定 eu-doc 是否能从 demo 变成真正可扩展的平台。

### 第二优先级：打通完整闭环

```text
1. 企业注册
2. 创建企业
3. 企业认证
4. 创建产品
5. 上传证书/DoC/说明书
6. 管理员审核
7. 前台搜索
8. 产品资料页展示
9. 用户举报反馈
```

原因：第二阶段 10-20 家企业测试时，最需要验证的是这个闭环。

### 第三优先级：稳定性和迁移

```text
1. 环境变量
2. 小域名部署
3. HTTPS
4. 数据备份
5. 错误日志
6. 恢复测试
```

原因：用户后面一定会换服务器和域名，现在就要避免写死。

### 第四优先级：商业化预留

```text
1. plans 表
2. subscriptions 表
3. usage_records 表
4. 功能权限判断函数
```

注意：只预留，不接支付。

## 7. 后续 AI 工作注意事项

后续 AI 接手 eu-doc 时，必须理解：

1. 用户不是只想做一个好看的证书页面，而是要做长期可扩展的平台。
2. 当前阶段不急着收费，但必须预留商业化结构。
3. 当前最重要的是数据模型、权限、分类、文件类型、企业认证、上传责任链。
4. 不要只围绕 `certificates` 表继续堆字段，应逐步转向 `products + documents` 架构。
5. 第二阶段会有 10-20 家真实企业测试，所以第一阶段结束前必须能模拟完整企业流程。
6. 项目会从本地迁移到测试服务器，再迁移到正式服务器，不能硬编码域名、路径、上传目录。
7. 用户最终面向全球，所以多语言字段和国际化能力必须预留。
8. 支付、OCR、API、CDN 可以后移，但不要把结构设计死。
9. 修改前应先评估是否会破坏现有 demo，并尽量采用兼容式迁移。
10. 每次开发新功能后，应更新 TODO 状态、必要时更新 WORK_LOG。
