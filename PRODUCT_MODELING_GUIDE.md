# eu-doc 产品建模规则：产品系列、型号与文件覆盖范围

最后更新：2026-06-28 00:00:00 CST

## 核心结论

eu-doc 不应默认采用“一个型号 = 一个产品”的规则。

更通用、更适合认证文件平台的结构是：

- 公司
- 产品 / 产品系列
- 型号 / 变体
- 文件资料
- 文件覆盖范围

也就是说：

- 一个产品条目可以代表一个具体产品，也可以代表一个产品系列。
- 一个产品下可以有多个适用型号 / 变体。
- 一个证书、DoC 或说明书可以覆盖全部型号，也可以只覆盖部分型号。

## 为什么这样设计

在 CE、DoC、PPE、电子电器、玩具、机械等合规场景中，一个文件经常覆盖：

- 一个具体型号。
- 多个相近型号。
- 一个产品系列。
- 一个产品类型、批次或变体组合。

合规文件的重点不是“必须一个型号一份文件”，而是必须能清楚识别该文件覆盖哪些产品、型号、类型或批次。

因此系统必须支持灵活覆盖关系，不能把逻辑写死为：

- 一个证书 = 一个产品。
- 一个型号 = 一个产品。
- 多个型号一定拆成多个产品。
- 多个型号一定合并成一个产品。

## 当前阶段采用的轻量规则

当前阶段先不大改数据库为复杂关系表，先采用轻量版：

1. `products` 表仍表示“产品 / 产品系列”。
2. `products.name` 用作产品名称或产品系列名称。
3. `products.model` 暂时用作主型号、系列名或型号列表摘要。
4. 批量导入识别到多个型号时，默认建议创建一个产品系列，而不是多个产品。
5. 文件暂时默认覆盖该产品条目下的所有适用型号。
6. 后续再升级为独立型号表和文件覆盖关系表。

## 推荐前端展示

产品详情页建议展示为：

- 产品名称：`Equestrian Helmet F20 Series`
- 分类：`个人防护用品 / 马术头盔`
- 适用型号：
  - `F20-201AL`
  - `F20-201AE`
  - `F20-201AE-MP`
  - `F20-202AL`
- 文件资料：
  - 资质证书：覆盖全部型号 / 覆盖部分型号
  - DoC 声明文件：覆盖全部型号 / 覆盖部分型号
  - 使用说明书：覆盖全部型号 / 覆盖部分型号

## 批量导入规则

批量导入时，AI / 系统应遵循：

1. 从文件名中尽量识别所有型号，而不是只取第一个型号。
2. 如果多个型号明显属于同一系列，默认建议创建一个产品系列。
3. 产品名称不要直接等于单个型号。
4. 可从文件名中提取更自然的产品名称，例如：
   - 文件名：`Helmet - F20-201AL,F20-201AE,F20-201AE-MP,F20-202AL - Equestrian helmet - EN.png`
   - 建议产品名称：`Equestrian Helmet F20 Series`
   - 适用型号：`F20-201AL, F20-201AE, F20-201AE-MP, F20-202AL`
5. 用户必须可以手动改成：
   - 作为一个产品系列整理。
   - 拆成多个产品。
   - 关联到已有产品。

## 什么时候合并成一个产品系列

建议合并：

- 用途相同。
- 外观/结构相近。
- 只是尺寸、颜色、配置、后缀不同。
- 共用同一份证书、DoC 或说明书。
- 用户搜索任一型号时，进入同一个资料页是合理的。

## 什么时候拆成多个产品

建议拆分：

- 用途不同。
- 结构明显不同。
- 认证标准不同。
- 证书、DoC、说明书完全不同。
- 用户通常会把它们视为不同商品。

## 后续专业版数据库目标

后续建议升级为：

- `products`：产品 / 产品系列。
- `product_models`：产品下的型号 / 变体。
- `documents`：证书、DoC、说明书等文件。
- `document_model_links`：文件覆盖哪些型号。

这样可以支持：

- 一个文件覆盖全部型号。
- 一个文件覆盖部分型号。
- 一个型号拥有多份文件。
- 一个产品系列下有多个型号。
- 搜索任意型号都能找到对应产品和文件。

## AI 修改本项目时必须遵守

AI 后续修改产品、批量导入、搜索、文件管理、证书页面时，不要再把“型号”简单等同于“产品”。

如果发现代码、文案或字段命名暗示“一个型号就是一个产品”，应优先改为“产品 / 产品系列 + 适用型号”的表达。
