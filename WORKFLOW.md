# eu-doc 工作流规则

最后更新：2026-06-15 14:18:52 CST

本文件记录用户要求的 AI 协作工作流程。为避免 token 浪费，本项目只使用少量统一工作文件，不再分散生成多个重复的今日总结、明日计划、完成报告。

## 统一工作文件

- `AGENTS.md`：AI 如何读取项目、如何避免浪费 token。
- `CURRENT_STATUS.md`：当前项目状态短摘要。
- `PROJECT_STRUCTURE.md`：项目结构说明。
- `WORKFLOW.md`：本文件，记录工作流规则。
- `TODO.md`：当前任务清单。
- `WORK_LOG.md`：最近三天工作日志、每日总结、次日计划。

## 1. 版本号规则

阶段性修改应对应版本号，方便备份和回退。

不是每次小改都升级版本号。以下情况通常不需要升级版本：

- 修改错别字。
- 调整一句注释。
- 不影响功能的小文案。

以下情况应记录版本号：

- 新增功能。
- 修复影响使用的问题。
- 调整项目结构。
- 修改部署、数据、登录、安全、工作流规则。
- 完成一个阶段性任务。

当前版本规则沿用：

- 小版本：`1.0.1`、`1.0.2`、`1.0.3`
- 新功能：`1.1.1`、`1.2.1`

## 2. 工作日志规则

工作过程中不要每完成一个小步骤就写完整总结。

默认规则：

- 小步骤完成：只在 `TODO.md` 更新任务状态，必要时在最终回复里简短说明。
- 阶段性成果完成：可以在 `WORK_LOG.md` 追加一条简短记录。
- 每日完整总结：只有用户明确要求“总结今天工作”时才写入 `WORK_LOG.md`。

工作日志记录时必须包含：

- 修改时间，精确到秒。
- 关联任务编号。
- 关联版本号。
- 修改了什么。
- 是否完成。
- 还有什么待完善。

时间格式：

```text
YYYY-MM-DD HH:mm:ss Z
```

## 3. TODO 规则

开始一个项目或阶段任务时，应在 `TODO.md` 建立任务清单。

任务应包含：

- 任务编号。
- 任务名称。
- 优先级。
- 状态。
- 创建时间。
- 完成时间。
- 关联版本。
- 完成内容。
- 待完善内容。

任务状态统一使用：

- `未开始`
- `进行中`
- `待复查`
- `已完成`
- `需改进`
- `已归档`

如果任务没有完全完善，不要标记为已完成，应标记为 `待复查` 或 `需改进`，并说明还差什么。

## 4. 每日总结规则

当用户要求“总结今天工作”时，AI 应把当天工作汇总写入 `WORK_LOG.md`。

每日总结应包含：

- 当天具体做了什么。
- 对应时间。
- 完成了什么。
- 还有什么没完善。
- 是否需要用户确认。

不要再额外新建多个 `DAILY_SUMMARY_xxx.md`、`WORK_SUMMARY_xxx.md`，除非用户明确要求导出单独文件。

## 5. 次日计划和继续工作规则

每日总结后，应在 `WORK_LOG.md` 的“下一步计划”部分写明下一次工作方向。

当用户说“继续工作”时，AI 应主动读取：

1. `AGENTS.md`
2. `CURRENT_STATUS.md`
3. `TODO.md`
4. `WORK_LOG.md` 最近三天内容

用户不需要重新解释上次做到哪里。

## 三天保留规则

根目录可以保留最近三天的工作类文档。

超过三天的工作类文档应移动到 `docs/archive/`。

未来优先使用 `WORK_LOG.md` 记录最近三天内容，避免继续生成多个重复总结文件。

## 6. 前一天总结/计划的读后归档规则

每日结束时，只有在用户明确要求总结时，AI 才把当日总结和次日计划写入 `WORK_LOG.md`。

第二天用户开启新对话并说“继续工作”时，AI 应：

1. 读取 `AGENTS.md`、`CURRENT_STATUS.md`、`WORKFLOW.md`、`TODO.md`、`WORK_LOG.md`。
2. 理解前一天完成了什么、还有什么没完成、下一步做什么。
3. 在开始当天新工作前，把已经读取并吸收的前一天总结/计划视为历史上下文。
4. 若存在单独的前一天总结/计划文件，应移动到 `docs/archive/` 对应目录。
5. 当天继续工作时，只维护必要的 TODO 状态；直到用户要求总结时，再在 `WORK_LOG.md` 中生成当日总结和下一步计划。

原则：

- 根目录只保留当前仍需要 AI 默认读取的少量工作文件。
- 已经被 AI 读取并用于接续工作的旧总结/计划，应归档，不要长期留在根目录。
- 历史文件不删除，只移动到 `docs/archive/`。


## 7. 避免频繁总结规则

用户希望一天持续工作，到当天最后由用户主动要求总结。

因此 AI 不应在每次小修改后自动写完整总结，也不应频繁更新 `WORK_LOG.md`。

推荐节奏：

- 工作中：专注完成任务。
- 小修改后：只在对话中简短说明，不写入长日志。
- 阶段性完成后：更新 `TODO.md` 状态。
- 用户说“总结今天工作”时：一次性汇总当天所有工作，写入 `WORK_LOG.md`，并规划下一步。

这样可以减少重复文档和 token 消耗。
