# eu-doc 工作流规则

最后更新：2026-06-16 23:17:08 CST
当前规则版本：v1.6.1

本文件是 eu-doc 的“工作方式总规则”。目标是：让 AI 少读无关文件、少生成重复文档、每个重要阶段都有可回退备份。

## 统一工作文件

根目录默认只保留这些常用工作文件：

- `AGENTS.md`：AI 阅读范围规则，避免浪费 token。
- `CURRENT_STATUS.md`：当前项目状态短摘要。
- `PROJECT_STRUCTURE.md`：项目结构说明。
- `WORKFLOW.md`：本文件，记录工作流、版本和备份规则。
- `TODO.md`：当前任务清单。
- `WORK_LOG.md`：最近三天工作日志、用户要求时的每日总结、下一步计划。

除非用户明确要求，不再新建多个 `DAILY_SUMMARY_xxx.md`、`FINAL_REPORT_xxx.md`、`TOMORROW_PLAN_xxx.md` 这类重复文件。

## 1. 版本号规则

版本号是“这个阶段成果的名字”，Git 提交是“可回退的备份点”。两者要配合使用。

当前版本从 `v1.6.1` 继续。历史上已经出现的 `1.1.0`、`1.4.0`、`1.6.0` 等记录不强行改名，避免破坏历史；从现在起统一使用下面规则。

### 什么时候升级版本

需要升级版本：

- 新增功能。
- 修复影响使用的问题。
- 调整项目结构。
- 修改部署、登录、安全、数据库、工作流规则。
- 完成一个可以独立回退的阶段性任务。

通常不升级版本：

- 改错别字。
- 改一句注释。
- 不影响功能的小文案。
- 工作中临时试错但没有形成稳定结果。

### 怎么升级版本

- 小修复/小优化：最后一位 +1，例如 `v1.6.1` -> `v1.6.2`。
- 用户说“开始进行下一个功能”或完成新的主要功能：中间一位 +1，最后一位从 1 开始，例如 `v1.6.2` -> `v1.7.1`。
- 大版本暂不使用，除非以后整体重构或 v2 项目正式上线。

### 每次版本升级必须同步更新

只要升级版本，必须同时检查这四处：

1. `.version`：写入当前版本，例如 `v1.6.1`。
2. `TODO.md`：对应任务写入关联版本和完成状态。
3. `WORK_LOG.md`：只在阶段性成果或用户要求总结时记录。
4. Git 提交信息：包含版本号和任务编号，例如 `v1.6.1 EU-T026 修正版本和备份规则`。

## 2. 备份规则

备份分三层，不能混淆：

- 本地文件：正在编辑的内容，还不算可靠备份。
- Git commit：本机可回退的备份点。
- GitHub push：云端备份，电脑出问题也能找回。

### 什么时候提交 Git commit

以下情况必须提交一次本地备份：

- 完成一个 TODO 小任务。
- 修复一个明确 bug。
- 改完一组相关 UI/功能。
- 更新版本号。
- 部署到服务器之前。
- 整理项目结构或工作流规则。

以下情况可以暂不提交：

- 只改错别字。
- 临时调试，还没有稳定结果。
- 同一个小任务还没完成，仍在连续修改。

### 什么时候推送 GitHub

以下情况必须确认并推送到 GitHub：

- 每天结束总结后。
- 部署到腾讯云之前。
- 一个重要功能完成并验证通过后。
- 用户明确说“备份到 GitHub”。

如果发现本地领先 GitHub 很多次提交，AI 应提醒用户：当前只有本地备份，GitHub 还不是最新备份。

### 推送前安全检查

推送 GitHub 前必须检查：

- 不提交 `.env`、token、服务器密钥、私钥、真实密码。
- 不提交 `node_modules/`。
- 不提交 `dist/`，除非部署流程明确需要。
- 不提交数据库文件。
- 不提交用户上传的真实证书文件。

如果不确定是否包含敏感信息，先只报告风险，不直接 push。

## 3. TODO 规则

`TODO.md` 是 AI 继续工作的主入口。任务编号像“身份证”，不能重复。

任务必须包含：

- 任务编号。
- 任务名称。
- 优先级。
- 状态。
- 创建时间。
- 完成时间。
- 关联版本。
- 待完善内容。

任务状态统一使用：

- `未开始`
- `进行中`
- `待复查`
- `已完成`
- `需改进`
- `已归档`

如果任务没有完全完善，不要标记为已完成，应标记为 `待复查` 或 `需改进`，并写清楚还差什么。

## 4. 工作日志和每日总结规则

工作中不要每完成一个小步骤就写完整总结，避免文档越来越多、token 越来越浪费。

默认节奏：

- 工作中：专注完成任务。
- 小修改后：只在对话里简短说明。
- 阶段性完成后：更新 `TODO.md`，必要时追加一条简短 `WORK_LOG.md`。
- 用户说“总结今天工作”时：再把当天工作、时间、完成内容、未完成内容、明日计划写入 `WORK_LOG.md`。

工作日志记录时必须包含：

- 时间，精确到秒，格式 `YYYY-MM-DD HH:mm:ss Z`。
- 关联任务编号。
- 关联版本号。
- 修改了什么。
- 是否完成。
- 还有什么待完善。

## 5. 继续工作规则

当用户说“继续工作”时，AI 应主动读取：

1. `AGENTS.md`
2. `CURRENT_STATUS.md`
3. `WORKFLOW.md`
4. `TODO.md`
5. `WORK_LOG.md` 最近三天内容

用户不需要重新解释上次做到哪里。

如果前一天有单独总结/计划文件，AI 读取并理解后，应移动到 `docs/archive/`，不要继续留在根目录造成重复读取。

## 6. 文档归档规则

根目录只放 AI 当前需要默认读取的少量文件。

历史总结、测试报告、部署报告、完成报告应归档到：

- `docs/archive/summaries/`
- `docs/archive/status-and-plans/`
- `docs/archive/reports/`
- `docs/archive/deployment/`
- `docs/archive/fixes/`
- `docs/archive/releases/`

历史文件不删除，只归档。AI 默认不读取 `docs/archive/`，除非用户要求追溯历史。

## 7. 部署前标准流程

部署到 christmasldw.com / 腾讯云之前，按顺序执行：

1. 本地测试通过。
2. 检查 Git 状态，确认没有意外文件。
3. 更新必要版本号、TODO、工作日志。
4. 创建 Git commit。
5. 推送 GitHub。
6. 再上传/部署服务器。
7. 部署后做线上验证。

原则：先有本地稳定版本和 GitHub 云端备份，再动线上环境。
